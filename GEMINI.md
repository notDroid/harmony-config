# Config Repository Agents Guide

This repository manages environment-specific configurations and deployments for Harmony Chat. It follows GitOps principles, acting as the environment-specific source of truth while the `app-repo` remains environment-agnostic.

## Stack
- **Task** (Taskfile.yaml)
- **Helmfile** (for templating manifests)
- **ArgoCD** (for GitOps continuous deployment)
- **SOPS** (for secrets management)
- **Kubernetes / Kind** (for local E2E testing)
- **Terraform / Terragrunt** (for AWS provisioning)

## Commands
```bash
# Initial Setup
task setup          # Creates local .env from .env.example

# Local E2E Testing (ArgoCD on Kind)
task local:up       # Runs the entire local E2E pipeline (Cluster + ArgoCD + OCI + Deploy)
task local:cluster  # Creates the local Kind cluster
task local:argocd   # Installs ArgoCD and creates namespaces
task local:deploy   # Applies the ArgoCD Application definition

# Infrastructure & OCI Pipeline
task k8s:fetch      # Fetches agnostic k8s components from app-repo into .cache/
task k8s:template   # Templates Helmfile to standard Kubernetes manifests in dist/
task k8s:publish    # Packages rendered manifests and pushes them to the OCI registry

# Cleanup
task clean          # Cleans up the dist/ and .cache/ directories
```

## Hard Limits
- **NEVER** commit unencrypted secrets. Always use SOPS for `secrets.yaml` files.
- **NEVER** place environment-specific logic or production values in the `app-repo`. All staging/production values, secrets, and environment overrides MUST live here in the `config-repo`.
- **NEVER** edit files in `.cache/k8s` or `dist/` directly. These are dynamically generated by `task k8s:fetch` and `task k8s:template`.

## Conventions
- **GitOps Separation:** 
  - The **App Repo** handles application source code, Docker image building, and generic infrastructure templates (as well as local-only `docker-compose` setups).
  - This **Config Repo** contains all specific configurations, SOPS-encrypted secrets, and triggers for deployments.
- **Dynamic Configuration:** The `config.yaml` file acts as the primary configuration for the project's global settings (project name, k8s version to fetch, registry URLs). The `Taskfile.yaml` dynamically populates its variables from this file using `yq`.
- **Helmfile Templating Strategy:** We use Helmfile to render standard Kubernetes manifests (`task k8s:template`) before packaging them as OCI artifacts (`task k8s:publish`). The `HELMFILE_VALUES` combine generic values from the fetched k8s components (in `.cache/k8s/`) with environment-specific overrides defined in `environments/<env>/`.

## Gotchas
- **Cross-Repo Dependencies:** The `local:up` task implicitly depends on building the Docker images in the `app-repo`. It achieves this by navigating into the `app-repo` to run `task kind:load`. Ensure your local `app-repo` is up-to-date and accessible.
- **Ephemeral Cache:** The `.cache/k8s/` folder contains a downloaded snapshot of the `app-repo`'s k8s directory based on the `K8S_VERSION` defined in `config.yaml`. Do not try to make persistent changes here; update the `app-repo` instead.

## Structure Pointers
- `config.yaml`: Global configuration file (read by Taskfile).
- `environments/<env>/`: The absolute source of truth for environment-specific values and secrets.
  - `environments/local/`: Values, secrets, and ArgoCD definitions for the local Kind cluster.
  - `environments/staging/`: Values, secrets, and overrides for the remote staging environment.
- `.cache/k8s/`: Ephemeral directory containing fetched k8s components.
- `dist/`: Output directory for rendered Kubernetes manifests.
- `terraform/`: Contains Terragrunt configurations for provisioning AWS infrastructure.
